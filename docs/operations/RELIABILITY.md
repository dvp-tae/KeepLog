# 신뢰성 가이드

## 신뢰성 목표

- 콘텐츠 수정이 사이트 빌드를 깨뜨리지 않아야 한다
- UI 변경에는 반복 가능한 검증 경로가 있어야 한다
- CI는 타입, 린트, 포맷, 문서 링크, 빌드 오류를 잡아야 한다
- 배포는 저장소만으로 재현 가능해야 한다

## 자주 발생하는 실패 유형

- 문서 내부 상대 링크 깨짐
- 잘못된 Markdown 프론트매터 또는 예상 밖의 콘텐츠 형식
- 코드 리뷰만으로 놓치는 UI 회귀
- GitHub Pages 배포 후 정적 자산 경로 깨짐

## 복구 원칙

- 넓은 리팩터 전에 먼저 로컬에서 재현한다
- 실패한 명령, 페이지, 워크플로우 단계를 활성 실행 계획에 남긴다
- 루프를 복원할 수 있는 가장 좁은 수정부터 시도한다
- 배포가 실패하면 해결되거나 다음 수정으로 복구될 때까지 사용자 영향이 있는 장애로 본다

## 필수 검증

- 앱 또는 워크플로우 변경이 머지 직전이라면 `pnpm verify`
- UI 변경이면 브라우저 확인
- 머지 전 GitHub Actions 상태 녹색 확인
