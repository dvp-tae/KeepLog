# KeepLog 핵심 원칙

## 제품 포지셔닝

KeepLog는 개인 브랜딩과 기록을 동시에 다루는 공간이다. 역할은 크게 두 가지다.

- 읽기 쉬운 이력서와 포트폴리오를 통해 신뢰를 보여준다
- 차분하고 오래 읽히는 블로그를 통해 글을 발행한다

사이트는 복잡해 보이기보다 정돈되어 보여야 한다. 훑어보기와 깊게 읽기가 모두 가능해야 한다.

## UX 원칙

- 새로움보다 콘텐츠 명확성이 우선이다
- 내비게이션은 데스크톱과 모바일 모두에서 직관적이어야 한다
- 이력서, 포트폴리오, 블로그는 서로 다른 모드지만 하나의 제품처럼 느껴져야 한다
- 시각 시스템의 중심은 타이포그래피와 여백이며 장식은 보조 수단이다
- 애니메이션은 구조 변화나 상태 변화를 설명할 때만 사용한다

## 프론트엔드 아키텍처 원칙

- `blog/src/features/`의 feature-first 구조를 유지한다
- 콘텐츠 페이지 데이터 준비는 가능하면 서버 렌더링 쪽에 둔다
- Markdown 파싱과 콘텐츠 로딩 로직은 한곳에 모은다
- 일회성 UI 추상화보다 기존 공용 프리미티브 재사용을 우선한다

## 에이전트 작업 원칙

- 큰 재작성보다 작은 검증 가능한 수정이 더 안전하다
- 의미 있는 변경은 실행 계획이나 문서 갱신을 반드시 남긴다
- 브라우저 확인이 없으면 시각 작업은 완료로 보지 않는다
- 한 문단으로 설명하기 어려운 디자인 선택은 너무 암묵적일 가능성이 높다

## 콘텐츠 원칙

- `post/`의 Markdown 파일은 임시 노트가 아니라 실제 발행 콘텐츠다
- 프론트매터는 예측 가능하게 유지해야 에이전트가 정렬, 필터, export를 안전하게 수행할 수 있다
- `draft` 동작은 명시적으로 유지되어야 하며 리팩터링 중에도 깨지면 안 된다

## 전달 및 운영 원칙

- 배포 파이프라인이 깨지는 것은 제품 버그로 본다
- 문서 드리프트는 외형 문제가 아니라 신뢰성 문제다
- CI는 리뷰 전에 흔한 실패 유형을 먼저 잡아야 한다
