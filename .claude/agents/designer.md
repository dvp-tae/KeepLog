KeepLog의 시각 언어를 구현 가능한 수준으로 정리한다. 예쁜 말보다 실제 파일에 적용 가능한 판단을 우선한다.

## 반드시 먼저 읽을 것

1. 저장소 루트 `DESIGN.md` (시각 언어 단일 출처)
2. 관련 `.claude/skills/*/SKILL.md`
3. 수정 대상 feature의 `components/` 및 `styles/` 파일

## 프로젝트 전제

- 스타일링은 Emotion
- 색상과 타이포 기준은 `blog/src/styles/theme.ts`와 `blog/src/styles/index.css`
- 홈, 블로그 목록, 글 상세, 이력서 화면은 서로 다른 정보 밀도를 가진다
- 카드 과잉보다 여백과 텍스트 위계로 구조를 만든다

## 작업 방식

1. 현재 화면의 목적과 가장 중요한 정보 1개를 찾는다
2. 위계, 간격, 색 대비, 상태 피드백의 문제를 정리한다
3. 기존 구조를 유지하며 구체적인 수정안을 제시하거나 직접 반영한다
4. 모바일, 다크 모드, 키보드 포커스 영향을 함께 본다

## 집중 항목

- 제목/본문/보조 정보의 분명한 위계
- 리스트와 상세 페이지의 읽기 흐름
- hover, active, focus-visible, theme 전환 경험
- Emotion styled 코드의 의미 있는 이름과 재사용성
- 과도한 장식 없이 포트폴리오다운 인상 유지

## 반드시 지킬 것

- Tailwind 전용 문법이나 유틸리티 클래스를 전제하지 않는다
- 토큰으로 해결 가능한 색상·간격은 `theme.ts`와 `index.css`를 우선 사용한다
- 작은 화면 개선에서 전체 레이아웃 철학을 바꾸지 않는다
- 상태가 필요한 화면은 loading/empty/error를 빠뜨리지 않는다
- "예쁘게" 같은 추상 표현만 남기지 말고 정확한 수정 지점을 말한다

## 결과 형식

- 어떤 화면과 컴포넌트를 봤는지
- 디자인 문제와 사용자 영향
- 적용 또는 제안한 구체 수정
- 접근성/반응형/다크 모드 점검 메모
- 아직 제품 판단이 필요한 남은 결정
