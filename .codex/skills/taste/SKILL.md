# KeepLog Taste Skill

신규 섹션, 신규 페이지, 혹은 인상 개선이 필요한 UI 작업에 사용합니다.

## 먼저 확인할 문서

1. `.codex/DESIGN.md`
2. 현재 수정할 feature의 컴포넌트, 스타일, 데이터 경로

`DESIGN.md`와 충돌하면 항상 `DESIGN.md`를 우선합니다.

## 기본 설정값

- 톤: 정제된 포트폴리오 + 읽기 쉬운 블로그
- 강도: 과장된 쇼케이스보다 차분한 인상 개선
- 구조: feature-first, Emotion styled, theme 토큰 우선

## 핵심 규칙

### 의존성 확인

서드파티 라이브러리를 import하기 전에 반드시 `blog/package.json`을 확인합니다.
존재하지 않으면 추가 설치가 필요한지 먼저 판단하고, 불필요하면 기존 스택으로 해결합니다.

### 절대 하지 말 것

- Tailwind 전용 클래스 규칙을 Emotion 코드에 복붙
- 토큰으로 해결 가능한 값을 임의 hex/px로 반복 작성
- 작은 개선 작업을 이유로 화면 전체를 재구성
- `console.log`, 임시 주석, `any` 타입 남기기
- 의미 없는 장식 모션 추가

### 인터랙션 상태

상태가 존재하는 화면은 `loading / empty / error`를 고려합니다.
정적 페이지라면 모든 상태를 억지로 만들 필요는 없지만, 데이터 로딩이나 실패 가능성이 있는 지점은 예외 없이 검토합니다.

- `Loading`: 기존 레이아웃을 해치지 않는 스켈레톤 또는 자리표시 상태
- `Empty`: 사용자가 다음 행동을 이해할 수 있는 안내
- `Error`: 실패 원인과 다음 행동이 드러나는 메시지

### 성능 가드레일

- 애니메이션은 `transform`, `opacity`, `box-shadow` 중심으로 제한합니다.
- Client Component가 꼭 필요한 인터랙션만 클라이언트로 둡니다.
- 서버 컴포넌트에서 해결 가능한 데이터/정적 렌더링은 유지합니다.

## KeepLog 전용 작업 흐름

1. 현재 화면의 목적을 한 문장으로 정리합니다.
2. 이 화면에서 가장 중요한 정보 1개와 보조 정보 1개를 구분합니다.
3. `theme.ts`와 `index.css` 안에서 재사용 가능한 토큰을 먼저 찾습니다.
4. 레이아웃, 타이포, 인터랙션을 최소 수정으로 개선합니다.
5. 모바일과 다크 모드 영향을 같이 점검합니다.

## 구현 기준

- 홈 화면은 첫인상과 탐색 진입이 동시에 보여야 합니다.
- 블로그 목록은 스캔 속도와 읽기 흐름이 우선입니다.
- 글 상세는 마크다운 가독성과 코드 블록 대비가 우선입니다.
- 이력서 화면은 많은 정보량 속에서도 위치 감각을 잃지 않아야 합니다.

## 권장 패턴

- 카드 대신 여백, 분리선, 명확한 타이포 위계로 구조를 만듭니다.
- 링크/버튼 hover는 짧고 가벼운 피드백으로 끝냅니다.
- 제목 계층은 강하게, 보조 설명은 절제해서 씁니다.
- 배경은 완전한 평면보다 미세한 톤 변화가 있을 수 있지만 과해서는 안 됩니다.

## 사용 예시

```text
/project:taste KeepLog 홈 화면 hero 영역을 더 정제된 포트폴리오 톤으로 개선해줘
/project:taste 블로그 목록의 hover 피드백과 타이포 위계를 다듬어줘
/project:taste 이력서 페이지의 섹션 헤더를 더 읽기 쉽게 정리해줘
```
