# KeepLog Redesign Skill

기존 화면을 완전히 갈아엎지 않고, 현재 스택과 구조를 보존하면서 개선할 때 사용합니다.

## 먼저 확인할 문서

1. `.codex/DESIGN.md`
2. 수정 대상 feature의 view, styles, model 파일
3. `blog/package.json`

규칙 충돌 시 `DESIGN.md`를 우선합니다.

## 작업 순서

1. **Scan** — 현재 화면 목적, 데이터 흐름, 스타일 작성 방식을 파악
2. **Diagnose** — 시각/구조 문제를 항목별로 정리
3. **Fix** — 기존 스택을 유지하며 필요한 부분만 수정
4. **Verify** — 린트, 타입, 실제 화면 영향을 최소 범위로 확인

## 개선 우선순위

1. **정보 위계 복구** — 지금 무엇을 먼저 읽어야 하는지 명확히 만듭니다.
2. **타이포 정리** — 제목, 본문, 보조 정보의 역할 분리를 회복합니다.
3. **간격과 폭 조정** — 읽기 흐름을 방해하는 넓이/패딩/정렬 문제를 고칩니다.
4. **상호작용 피드백 보강** — hover, active, focus, theme 전환 감각을 정리합니다.
5. **색상 사용 정리** — 토큰 범위 안에서 대비와 브랜드 포인트를 정돈합니다.
6. **상태 처리 보강** — empty/error/loading이 필요한 지점만 정확히 넣습니다.
7. **코드 정리** — 반복 스타일, 하드코딩, 불필요한 복잡도를 줄입니다.

## 주요 감사 항목

- 타이포그래피: 반복되는 임의 숫자, 불명확한 제목 계층, 본문 가독성 저하
- 컬러와 서피스: 토큰 밖 색상 남용, 라이트/다크 대비 저하
- 레이아웃: 과한 카드화, 좁거나 넓은 읽기 폭, 모바일 여백 붕괴
- 인터랙션: hover만 있고 focus가 없거나, 모션이 과하거나 어색한 경우
- 상태 표현: 데이터가 없거나 실패했을 때 침묵하는 화면
- 코드 품질: feature 경계 침범, 중복 스타일, 의미 없는 wrapper 증가

## 규칙

- 프레임워크나 스타일링 방식을 교체하지 않습니다.
- 기존 기능과 정보 구조를 깨지 않는 범위에서 수정합니다.
- 새 라이브러리 import 전 `blog/package.json` 확인은 필수입니다.
- `theme.ts`와 전역 CSS 토큰으로 해결 가능한 값은 우선 재사용합니다.
- 수정은 리뷰 가능한 단위로 유지합니다.
- 재작성보다 국소 개선을 기본값으로 둡니다.

## KeepLog 전용 진단 질문

- 홈 화면에서 자기소개와 글 탐색 동선이 동시에 읽히는가
- 블로그 목록에서 포스트 제목과 태그가 한 눈에 구분되는가
- 상세 본문에서 마크다운 요소가 읽기 흐름을 깨지 않는가
- 이력서 페이지에서 현재 위치와 섹션 전환이 자연스러운가
- 다크 모드에서 코드 블록, 링크, 태그 색이 충분히 구분되는가

## 수정 원칙

- 스타일 수정은 가능하면 해당 feature의 `styles/*.ts`에 모읍니다.
- 공통 규칙이면 `src/styles/index.css` 또는 공용 컴포넌트로 올립니다.
- 데이터 가공 로직은 `model/*.server.ts` 또는 모델 파일에 남깁니다.
- 화면 개선 중 구조가 바뀌면 server/client 경계가 바뀌는지 같이 확인합니다.

## 사용 예시

```text
/project:redesign KeepLog 홈 화면을 재작성하지 말고 여백과 타이포만 정리해줘
/project:redesign 블로그 목록 페이지의 포스트 행 인터랙션을 더 차분하게 다듬어줘
/project:redesign 이력서 페이지에서 모바일 간격과 고정 내비게이션 동작을 점검해줘
```
